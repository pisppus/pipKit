# Debug Metrics Status Bar Mode - Requirements

## Feature Overview
Добавление нового режима отладки для pipGUI, который заменяет содержимое статус-бара на метрики производительности: загрузку CPU и использование RAM. Это дополнение к существующему режиму dirty rect debug overlay.

## User Stories

### 1. Режим отладки с метриками производительности
**As a** разработчик pipGUI приложений  
**I want** видеть метрики CPU и RAM в статус-баре  
**So that** я могу отслеживать производительность и выявлять утечки памяти в реальном времени

### 2. Переключение между режимами отладки
**As a** разработчик  
**I want** иметь возможность включать/выключать разные режимы отладки независимо  
**So that** я могу использовать dirty rect overlay и metrics status bar одновременно или по отдельности

### 3. Мониторинг загрузки CPU
**As a** разработчик  
**I want** видеть процент загрузки CPU для отрисовки и работы библиотеки  
**So that** я могу оптимизировать производительность моего UI

### 4. Отслеживание использования памяти
**As a** разработчик  
**I want** видеть детальную информацию о свободной памяти  
**So that** я могу выявлять утечки памяти и фрагментацию heap

## Acceptance Criteria

### 1.1 Режим Debug Metrics в статус-баре
- Система должна поддерживать новый режим отладки `DebugMetricsStatusBar`
- Когда режим включен, статус-бар должен отображать метрики вместо обычного содержимого
- Режим должен включаться/выключаться независимо от dirty rect overlay
- Формат отображения должен быть компактным и читаемым

### 1.2 Измерение загрузки CPU
- Система должна измерять загрузку CPU используя метод на основе времени (time-based), аналогичный LVGL
- Метод измерения НЕ должен зависеть от FreeRTOS (для поддержки любых микроконтроллеров)
- Алгоритм: CPU% = (busy_time / total_elapsed_time) * 100
  - Измеряется время начала и конца работы библиотеки (render + flush)
  - Вычисляется процент от общего времени между обновлениями
- Измерение должно показывать процент времени, затрачиваемого на работу библиотеки pipGUI
- Для ESP32 dual-core: отображается общая загрузка (не разделяется по ядрам)
- Обновление метрик должно происходить каждые 100ms

### 1.3 Мониторинг памяти RAM
- Система должна отображать несколько метрик памяти для полноценной отладки:
  - **Total Free Heap** (`esp_get_free_heap_size()`) - общая свободная память
  - **Free Internal DRAM** (`heap_caps_get_free_size(MALLOC_CAP_8BIT | MALLOC_CAP_INTERNAL)`) - свободная внутренняя память
  - **Largest Free Block** (`heap_caps_get_largest_free_block(MALLOC_CAP_8BIT)`) - самый большой непрерывный блок (индикатор фрагментации)
  - **Minimum Free Heap** (`esp_get_minimum_free_heap_size()`) - минимальное свободное за всё время (для выявления утечек)
- Все метрики должны отображаться в килобайтах в компактном формате
- Формат: `"CPU:xx% T:xxk D:xxk L:xxk M:xxk"` где T=Total, D=DRAM, L=Largest, M=Min

### 1.4 Интеграция с существующей Debug системой
- Новый функционал должен быть добавлен в существующий класс `Debug` в `lib/pipSystem/pipGUI/core/Debug.hpp`
- Должны использоваться существующие структуры `DebugMetrics` и методы `update()`, `formatStatusBar()`
- API должен быть согласован с существующим dirty rect debug функционалом
- Не должно быть конфликтов между разными режимами отладки

### 1.5 Формат отображения метрик
- Формат: `"CPU:xx% T:xxk D:xxk L:xxk M:xxk"` где:
  - CPU:xx% - процент загрузки процессора
  - T:xxk - Total free heap (общая свободная память в KB)
  - D:xxk - DRAM free (внутренняя память в KB)
  - L:xxk - Largest block (самый большой блок в KB)
  - M:xxk - Min free (минимум за всё время в KB)
- В режиме debug metrics статус-бар должен показывать только эти метрики
- Все остальные элементы статус-бара (иконки, батарея, время и т.д.) должны быть скрыты
- Остаётся только чистая основа статус-бара с метриками

### 1.6 API для управления режимом
- `Debug::setMetricsStatusBarEnabled(bool)` - включить/выключить режим
- `Debug::isMetricsStatusBarEnabled()` - проверить состояние режима
- Метод `formatStatusBar()` уже существует и должен использоваться для форматирования

## Technical Constraints

### Platform Requirements
- ESP32 (single-core и dual-core)
- FreeRTOS
- Arduino framework
- Существующая pipGUI архитектура

### Performance Requirements
- Измерение метрик не должно значительно влиять на производительность
- Обновление метрик должно происходить каждые 100ms
- Форматирование строки статус-бара должно быть быстрым (< 1ms)
- Измерение времени должно использовать высокоточный таймер (micros() или аналог)

### Memory Requirements
- Минимальное использование дополнительной памяти
- Использование существующих структур данных где возможно

## Dependencies
- Существующий класс `Debug` в `lib/pipSystem/pipGUI/core/Debug.hpp` и `Debug.cpp`
- ESP-IDF функции: `esp_get_free_heap_size()`, `heap_caps_get_free_size()`, `heap_caps_get_largest_free_block()`, `esp_get_minimum_free_heap_size()`
- FreeRTOS task statistics API для измерения CPU load
- Класс `GUI` для интеграции с статус-баром

## Out of Scope
- Графическое отображение метрик (графики, диаграммы)
- Логирование метрик в файл или по сети
- Настройка формата отображения через конфигурацию
- Отображение метрик в других местах кроме статус-бара

## Resolved Questions
1. ✅ Режим metrics status bar автоматически скрывает обычное содержимое статус-бара - остаётся только чистая основа с метриками
2. ✅ Не нужна возможность выбора метрик - всегда показываются CPU% и RAM
3. ✅ Метрики обновляются с фиксированной частотой 100ms
4. ✅ Callback для кастомного форматирования не нужен

## Implementation Notes
- Текущая реализация использует FreeRTOS idle tasks для измерения CPU - это нужно заменить на time-based метод
- Time-based метод: измеряем время работы библиотеки (от начала render до конца flush) и делим на общее прошедшее время
- Это даст независимость от RTOS и будет работать на любом микроконтроллере с функцией получения времени
